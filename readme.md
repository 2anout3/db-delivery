# Delivery Service

Веб-приложение для оформления и сопровождения доставок между филиалами службы доставки.

Проект состоит из backend API, статического frontend-интерфейса и PostgreSQL-базы данных, в которой хранится основная бизнес-модель: пользователи, клиенты, сотрудники, филиалы, доставки, курьерские заявки, статусы и история изменений.

## Стек проекта

### Backend

- **Python**
- **FastAPI** - HTTP API и раздача статического frontend.
- **SQLAlchemy** - ORM-модели и работа с PostgreSQL.
- **Pydantic** - схемы входных и выходных данных API.
- **Uvicorn** - ASGI-сервер для запуска приложения.
- **python-jose** - JWT-токены.
- **passlib** - хеширование и проверка паролей.

### Database

- **PostgreSQL**
- **PL/pgSQL**
- SQL-функции для расчета стоимости.
- SQL-процедуры для смены статусов и назначения курьера.
- Триггеры для автоматического расчета цены, фиксации даты доставки и записи истории статусов.
- Constraints и foreign keys для защиты целостности данных.

### Frontend

- **HTML**
- **CSS**
- **JavaScript ES modules**
- `fetch` для запросов к API.
- `localStorage` для хранения JWT-токена и профиля пользователя.

## Как запустить проект

Установить зависимости:

```bash
pip install -r requirements.txt
```

Создать пользователя и базу данных PostgreSQL, если они еще не созданы:

```bash
sudo -u postgres psql
```

```sql
CREATE USER dbproj WITH PASSWORD 'db123';
CREATE DATABASE delivery_service OWNER dbproj;
\q
```

Инициализировать базу:

```bash
psql -U dbproj -d delivery_service -f backend/init_db.sql
```

Запустить приложение:

```bash
uvicorn backend.main:app --reload
```

Открыть:

```text
http://127.0.0.1:8000
```

## Как сбросить базу данных

Скрипт `backend/init_db.sql` сам удаляет старые таблицы и создает их заново:

```bash
psql -U dbproj -d delivery_service -f backend/init_db.sql
```

Если нужно полностью пересоздать базу:

```bash
sudo -u postgres psql
```

```sql
DROP DATABASE IF EXISTS delivery_service;
CREATE DATABASE delivery_service OWNER dbproj;
\q
```

Затем снова:

```bash
psql -U dbproj -d delivery_service -f backend/init_db.sql
```

## Роли пользователей

### Клиент

Клиент может:

- зарегистрироваться;
- войти в систему;
- создавать доставки;
- смотреть свои входящие и исходящие доставки;
- запрашивать курьера для доставки до получателя.

### Оператор

Оператор выполняет операционную работу по доставкам:

- регистрирует доставку после создания клиентом;
- переводит посылку в статусы филиалов;
- отправляет посылку в путь;
- завершает доставку;
- управляет клиентами;
- видит рабочие доставки.

Отдельной роли менеджера в проекте больше нет: все его возможности переданы оператору.

### Курьер

Курьер не назначается вручную оператором. Он сам берет доступные курьерские заявки.

Курьер может взять заявку, если:

- заявка относится к его городу;
- заявка относится к его филиалу;
- вес доставки не превышает грузоподъемность курьера;
- доставка находится в подходящем статусе:
  - `Ожидает курьера` для забора у отправителя;
  - `В филиале назначения` для доставки получателю.

Курьер может менять только свои курьерские статусы:

- `Курьер получил посылку`;
- `Курьер доставляет получателю`;
- `Курьер доставил посылку`.

### Администратор

Администратор может:

- создавать аккаунты;
- видеть все доставки;
- создавать доставки от имени клиентов;
- выполнять системное управление.

## Статусы доставки

Текущая модель статусов:

| Код | Название | Кто выставляет | Назначение |
|---|---|---|---|
| `CREATED` | Создана | Клиент/админ при создании | Начальный статус доставки. |
| `REGISTERED` | Зарегистрирована | Оператор | Доставка принята в работу. |
| `WAITING_COURIER` | Ожидает курьера | Оператор | Нужен курьер для забора у отправителя. |
| `COURIER_ASSIGNED` | Курьер назначен | Курьер через самовзятие заявки | Курьер взял заявку. |
| `COURIER_PICKED_UP` | Курьер получил посылку | Назначенный курьер | Курьер забрал посылку у отправителя. |
| `AT_ORIGIN_BRANCH` | В филиале отправления | Оператор | Посылка находится в филиале отправления. |
| `IN_TRANSIT` | В пути | Оператор | Посылка едет между филиалами. Используется, если филиалы разные. |
| `AT_DESTINATION_BRANCH` | В филиале назначения | Оператор | Посылка прибыла в филиал назначения. |
| `COURIER_DELIVERING_RECIPIENT` | Курьер доставляет получателю | Назначенный курьер | Курьер начал доставку от филиала до получателя. |
| `COURIER_DELIVERED` | Курьер доставил посылку | Назначенный курьер | Курьер передал посылку получателю. |
| `DELIVERED` | Доставлена | Оператор | Финальное подтверждение доставки. |
| `CANCELLED` | Отменена | Оператор/админ в допустимых состояниях | Доставка отменена. |

## Основной сценарий доставки

1. Клиент создает доставку.
2. Система ставит статус `Создана`.
3. Оператор переводит доставку в статус `Зарегистрирована`.
4. Если нужен курьер для забора у отправителя:
   - оператор ставит `Ожидает курьера`;
   - курьер сам берет заявку;
   - система переводит доставку в `Курьер назначен`;
   - курьер ставит `Курьер получил посылку`;
   - оператор ставит `В филиале отправления`.
5. Если курьер для забора не нужен:
   - оператор сразу ставит `В филиале отправления`.
6. Если филиалы отправления и назначения разные:
   - оператор ставит `В пути`;
   - затем `В филиале назначения`.
7. Если филиалы совпадают:
   - оператор может перевести из `В филиале отправления` сразу в `В филиале назначения`.
8. Если получателю нужна курьерская доставка:
   - получатель создает курьерскую заявку;
   - курьер сам берет ее;
   - курьер ставит `Курьер доставляет получателю`;
   - затем `Курьер доставил посылку`;
   - оператор завершает доставку статусом `Доставлена`.
9. Если получатель забирает посылку из филиала:
   - оператор из статуса `В филиале назначения` ставит `Доставлена`.

## SQL-функции, процедуры и триггеры

### `calculate_delivery_cost`

Рассчитывает стоимость доставки.

Логика:

- если отправление и получение в одном городе, базовая цена фиксированная;
- если города разные, цена зависит от расстояния, веса и объявленной стоимости;
- `extra_cost` считается как процент от объявленной стоимости;
- `total_cost = base_cost + extra_cost`.

### `set_delivery_costs`

Триггерная функция, которая автоматически пересчитывает стоимость доставки перед вставкой или изменением важных полей.

### `set_delivery_timestamp`

Если доставка получает финальный статус `DELIVERED`, функция заполняет `delivered_at`.

Если статус откатывается с `DELIVERED`, дата доставки очищается.

### `write_delivery_status_history`

Автоматически пишет историю статусов в `delivery_status_history`.

### `validate_courier`

Проверяет, что запись в `couriers` создается только для активного сотрудника с ролью `courier`.

### `validate_courier_assignment`

Проверяет курьерскую заявку:

- курьер должен существовать и быть активным;
- нельзя создать две заявки одного типа для одной доставки.

### `change_delivery_status`

Главная процедура смены статуса доставки.

Она:

- проверяет, существует ли доставка;
- проверяет, существует ли новый статус;
- запрещает нелегальные переходы;
- записывает контекст изменения;
- обновляет текущий статус;
- через триггер пишет историю.

### `assign_courier_to_delivery`

Назначает курьера на курьерскую заявку.

Используется, когда курьер сам берет заявку.

Для `pickup`-заявки в статусе `WAITING_COURIER` процедура переводит доставку в `COURIER_ASSIGNED`.

- `backend/routers/auth.py` - вход, регистрация, профиль пользователя.

## Важные бизнес-правила

- Курьер не назначается оператором вручную.
- Курьер сам берет доступную заявку.
- Курьер может взять только заявку своего города и филиала.
- Вес доставки не должен превышать грузоподъемность курьера.
- Оператор выполняет все операционные действия по доставке вместо отдельного менеджера.
- Финальный статус `Доставлена` ставит оператор.
- Если получателю нужна курьерская доставка, финальный статус можно поставить только после статуса `Курьер доставил посылку`.
- Каждая смена статуса сохраняется в истории.
